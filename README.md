## Вопрос:
-  Был ли у тебя опыт полного ведения конкретной фичи (end-to-end или [feature owner](https://www.google.com/search?q=feature+owner))? 
Расскажи, как ты курировал процесс от обсуждения требований до выкатки в прод и что входило в твою зону ответственности, помимо написания кода

 - Какие по продолжительности фичи были на проекте? Практиковали декомпозицию крупных фич на подзадачи? Опиши принятый процесс работы над фичей от постановки до выпуска в релиз. 

## Ответ:

В целом основная часть (~60-70%) моей работы в последние 5 лет заключалась в проектировании и реализации нового функционала либо расширение старого.
Если говорить о процессе от идеи до релиза то его можно разделить на несколько этапов:

 1. Люди ответственные за генерирование бизнес идей формально описывают своё виденье и доносят его через Product Manager до непосредственных исполнителей (QA, дизайнеры, программисты) что-бы получить обратную связь. На этом этапе могут быть внесены ряд изменений в потенциальную фичу, пересмотрена механика работы, возможны принципиальные изменения в действия пользователя для достижение цели обсуждаемой фичи, возможен отказ от самой идеи.

 2. Когда стало ясно, что новой фиче быть и примерно понятно когда хотелось бы получить её реализацию начинается процесс планирования: __QA__ определяются с тест-кейсами, __программист__ который потенциально будет заниматься реализацией начинает процесс иследования и планирования.

    - На этапе исследования уточняются требования, определяется фронт работ (куда буду внесены изменения, структура модулей), подбираются инструменты которые могут потенциально помочь в реализации
    - По результатам исследования должен быть сформирован пул задач и установлены взаимные зависимости (некоторые задачи могут блокироваться другими задачами). При планировании желательно, что-бы задачи как можно меньше зависили друг от друга, это позволит распаралелить процесс реализации. Возможно появятся задачи, которые не относятся непосредственно к фиче, но блокируют реализацию части задачь (например, нужно внести изменения в деградировавший легаси)
    - Когда пулл задач ясен, производится оценка времени на реализацию
    - По результатам оценки задач могут быть скореектированы сроки в которые будет ожидаться фича

 3. Далее, определяемся с датой старта разработки и непосредственно кто будет заниматься разработкой. При разработке следует следовать принципу: "Чем раньше удалось выявить проблему, тем дешевле будет проблему решить", поэтому подключение QA на ранних этапах разработки для предварительного тестирования может существенно упростить разработку. В процессе реализации может выяснится, что требуются переоценка задачь (такое нередко бывает если работа ведётся со старым легаси), соответственно нужно завести нелбходимые задачи и оценить их с корректировкой сроков, так же возможен пересмотр деталей фичи.

      На практике хорошо себя показало использование каналов __Slack__. Под реализацию крупной фичи создаётся канал в __Slack__ и по мере разработки в него добовляются участники. На первоначальном этапе в канале будут разработчики фичи и Project Manager, когда часть работы будет сделана будут добавлены __QA__, для получения фидбеков при предварительном тестировании. В какой-то момент могут быть добавлены дизайнеры для согласования вопросов, могут быть добавлены непосредственные заказчики фичи для уточнений. В результате в канале будет подробный лог работы и контакты всех заинтересованных участников 


## Вопрос:

 - Кто оценивал задачи? Кто распределял задачи между разработчиками?
 - Что делали в ситуации, когда в процессе работы над задачей выяснялось что-то, что меняло ее оценку или scope?

## Ответ:
Для оценка задач обычно использовался __Planning poker__ при участии разных составов участников, в зависимости от того, что конкретно оценивается - если задача по реализации - то участники будут состоять из разработчиков и QA - разработчики оценивают сложность реализации, QA - тестирование. Если есть анамалии в оценках проводится обсуждение.

В ситуации если в процессе становится понятно, что требуется переоценка, то делается анализ, заводятся новые задачи, оцениватся и по результатам новых оценок принимается решение либо продолжать с корректировкой дедлайнов, либо фича ухожит в бэклог, а Project Manager сообщает заинтересованым участникам о необходимости корректировки планов


## Вопрос:

- Расскажи о кейсе в работе (интересная фича, сложный баг, оптимизация), которым ты можешь гордиться.

## Ответ:
  Из сложного - реализация растеризации деформации изображения по 4-м точкам (Для представления о задаче можно посмотреть реализацию на MSL в [моём репозитории](https://github.com/vitali-kurlovich/swift-image) )

Главная сложность заключалась в том, что реализация должна была работать на CPU на Main потоке, поскольку это было на устройствах младше iPhone 4, где GPU не поддерживал графических шейдеров. Пришлось написать на C реализацию програмной отрисовки непосредственно в буфер который использовался CGImage. Для оптимизации в интерактивном режиме использлвалась картинка меньшего разрешения. И ряд других приёмов оптимизации. 

Из сложных багов:  В программе был ф-л по хранению закладок на новости, закладки хранились на устройстве в CoreData хранилище которое синхронизировалось с сервером, через который проходила синхронизация с разными устройствами владельца. Это было во времена, когда только появился iCloud и заказчик захотел, что-бы закладки можно было редактировать offline c последующей синхронизацией изменений между устройствами. Естественно, серверная часть разрабатывалась без учёта такого ф-ла. По условию задачи нельзя было менять ни логику, ни модель данных на сервере, всё должно было быть реализованно на клиенте. Собственно, я занимался алгоритмом синхронизации и непосредственной реализацией. В процессе тестирования выяснилось, что синхронизация не работает. Моё первое предположение было в том, что поле UpdateDate на которое опирался алгоритм модифицируется на сервере. Собственно серверистам был задан вполне логический вопрос - не меняют ли они значение этого поля - и получен ответ: "Не меняем". Данный вопрос я через день повторил, поскольку ошибку в реализации я не нашёл - ответ со стороны глав-бекенд не изменился. После 3-х недель отладки и анализа пришлось снова обратится к бэкэнд-разработчику, что-бы тот показал как происходит сохранение данных на сервере - и в коде сохранения поле UpdateDate в момент отправки в базу данных выставлялось в текущую дату, что полностью ломало синхронизацию. Так что, самым сложным багом был баг, которого не было

Из архитектурных решений: Для банкинга настоял на изменении подхода к представлению и работой с данными. Приложение состояло из нескольких частей: Core - реализованная на Java часть, реализующая транспортный слой и слой по работе с данными, использовалась на сервере и Android напрямую, а на iOS, java-код конвертировался в C++ и через Obj-CPP велась работа с Core - обработка пакетов данных, получение данных форм, редактирование данных, всё это проходило через Core. Поскольку данные представляли собой reference типы всегда существовал риск случайно изменить данные которые менять не планировалось. Я настоял на том, что-бы сделать данные неизменяемыми, а при редактировании создавались новые данные с последущим явным замещением старых данных. В результате исчез целый класс трудновыявляемых багов
